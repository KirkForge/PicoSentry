# L3-SUS-001 — Dynamic Code Execution

**Rule ID:** L3-SUS-001  
**Backend:** Subprocess (universal)  
**Verdict:** DENY  
**Pattern:** `(?i)(eval\s*\(|exec\s*\(|compile\s*\()`

## Detection

Triggers when sandboxed output contains patterns indicating dynamic code execution:

- `eval(` — JavaScript/Python eval function
- `exec(` — Python exec function
- `compile(` — Python compile function

## Supply-Chain Relevance

Dynamic code execution is the #1 indicator of malicious post-install scripts. Attack patterns include:

- **event-stream** (2018): `eval()` used to decrypt and execute hidden payload
- **colors.js/faker.js** (2022): Runtime code injection via eval
- Obfuscated payloads commonly use `eval(atob('...'))` patterns

## False Positives

Legitimate uses of eval/exec:
- Build tools that eval configuration expressions
- Template engines that compile templates
- REPL/shell implementations

## Mitigation

If this rule triggers on legitimate code, consider:
1. Using `ast.literal_eval()` instead of `eval()` (Python)
2. Using `new Function()` instead of `eval()` (JavaScript)
3. Adding the pattern to a baseline allowlist
