# L3-SUS-002 — Shell Execution

**Rule ID:** L3-SUS-002  
**Backend:** Subprocess (universal)  
**Verdict:** DENY  
**Pattern:** `(?i)(subprocess|os\.system|os\.popen|commands\.getoutput)`

## Detection

Triggers when sandboxed output contains shell execution patterns:

- `subprocess` — Python subprocess module usage
- `os.system()` — Python system call
- `os.popen()` — Python popen call
- `commands.getoutput()` — Python commands module (legacy)

## Supply-Chain Relevance

Shell execution in post-install scripts is a common attack vector:

- **ua-parser-js** (2021): Post-install script spawned cryptocurrency miner via `child_process.exec()`
- **crossenv** (2017): Used subprocess to download and execute remote code
- Typosquat packages often use shell execution for data exfiltration

## False Positives

Common legitimate uses:
- Build scripts that compile native modules
- Test runners that spawn processes
- Package managers themselves

## Mitigation

1. Review all post-install/pre-install scripts before running
2. Use PicoDome's `strict` policy to block all process spawning
3. Consider the `node` or `python` policy for permissive but monitored execution
